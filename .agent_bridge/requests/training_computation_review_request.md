# [전체 시스템 연산 및 모델 학습 아키텍처 리뷰 요청]

현재 AirWriting 백엔드 서버 구동 중 다음과 같은 2가지 성능/경험적 이슈가 남아있습니다.
이번에는 **직접적인 코드 수정 대신, 아키텍처 및 훈련 파이프라인의 관점에서 해결 방향(제안사항)만 피드백**해 주시기 바랍니다.

## 🚨 사용자 리포트 (현재 발생하는 문제)
1. **성급한 인식 (False Positive / Premature Recognition)**
   - "글자 쓰기도 전에 인식해버림": 펜다운(`is_writing=True`) 상태가 되자마자 모델이 성급하게 특정 글자('D', 'A' 등)를 뱉어냅니다. 
   - 슬라이딩 윈도우 추론이 도입되면서 획의 극초반 부분(20프레임 내외, 약 0.2초)만 보고도 모델이 '신뢰도 100%'로 잘못된 글자를 확신해버립니다. (물론 펜을 떼면 올바른 글자로 오토코렉트 되지만, UI 경험상 매우 거슬립니다.)
2. **잔존하는 연산 버벅임 (Computation Bottleneck)**
   - 큐 사이즈 제한 해제 및 60Hz 주기 고정을 적용했음에도 간헐적인 버벅임이 남았습니다. 
   - 터미널 로그를 보면 WebSocket 연결 해제 에러(`ConnectionClosedError`)가 찍히며, AI 추론 로직이 비동기 이벤트 루프를 미세하게 블로킹하고 있을 확률이 높습니다.

## 🛠️ 클로드에게 요청하는 검토 및 제안 과제

1. **AI 모델 훈련(Training) 및 Partial Stroke 처리 방안 제안**
   - 현재 모델(`pure_bilstm.pt`)이 잘려진 부분 궤적(Partial Stroke)만 보고도 높은 신뢰도를 출력하는 **Overconfidence(과적합) 문제**가 있습니다.
   - 향후 재학습 시 Partial Data를 Augmentation으로 추가하는 방안, CTC Blank Label 활용, 혹은 예측 신뢰도(Confidence Threshold)를 엄격하게 제한하는 방안 등 **학습 파이프라인 개선 아이디어**를 제안해 주세요.
2. **연산(Computation) 병목 및 GIL 문제 해결 아키텍처 제안**
   - 현재 `loop.run_in_executor`를 사용해 백그라운드 스레드로 PyTorch 추론을 넘겼으나, 파이썬의 GIL(Global Interpreter Lock) 특성상 무거운 텐서 연산이 메인 UDP/WebSocket 비동기 루프에 잔렉을 유발할 수 있습니다.
   - 이를 근본적으로 해결하기 위한 **아키텍처(예: 멀티프로세싱 `ProcessPoolExecutor` 전환, C++ 기반 ONNX Runtime 도입, 분리된 추론 마이크로서비스 등)**를 제안해 주세요.
3. **슬라이딩 윈도우 파라미터 최적화 제안**
   - 현재 "버퍼 20프레임 이상일 때, 40프레임(약 0.5초) 주기"로 부분 추론을 돌리고 있습니다. 
   - 사용자 체감을 좋게 유지하면서 초기 노이즈 인식을 막기 위한 최적의 슬라이딩 윈도우 사이즈와 인터벌 설계에 대해 조언해 주세요.

**⚠️ 주의**: 이 문서에 대한 답변은 구체적인 `main.py` 코드 작성(Patch)보다는, 다음 개발 단계를 위한 **기술적 방향성 제안과 아키텍처 설계(Proposals)** 위주로 도출해 주시기 바랍니다.
